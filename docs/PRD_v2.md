# 工地照片 AI 整理系統 — 需求規格書(修訂版 v2)

> 用途:直接餵給 Claude Code / AI 開發工具使用。
> 修訂重點:訊息通道決策、改善前後照片關聯、回報人記錄、相簿多圖處理、AI 追問機制、分版開發、資料外傳聲明。

---

## 一、背景與目標

建設公司工地每日產生大量記錄照片(品質記錄、缺失待改、改善完成)。人工整理耗時且缺失追蹤容易漏勾稽。本系統讓工地人員在通訊軟體中直接傳送「照片 + 文字/語音說明」,由 AI 自動結構化為工地紀錄與缺失追蹤資料,並提供後台查詢、狀態管理與 Excel 匯出。

成功標準:
- 回報一筆缺失(拍照+說明)耗時 < 30 秒
- AI 欄位辨識準確率 > 85%(以人工確認修正率回推)
- 缺失從「回報 → 改善 → 銷案」全程可追蹤,不漏件

---

## 二、關鍵決策(開發前必須確認)

### 決策 1:訊息通道 — Telegram 或 LINE

| | Telegram | LINE |
|---|---|---|
| 台灣工地使用習慣 | 需另外安裝,推行阻力大 | 人人都有 |
| API 費用 | 完全免費 | 回覆免費,主動推播有額度限制 |
| 部署需求 | 可用 long polling,**不需公開網址** | 必須 webhook + 公開 HTTPS |
| 檔案取得 | 不限時 | 圖檔有取檔時效,須即時下載 |

**建議**:
- 使用者只有 1-3 人(特助自己用)→ 用 **Telegram**,開發最快
- 要全工地人員回報 → 必須用 **LINE**

**架構要求**:無論選哪個,訊息接收層必須寫成 **adapter 介面**(`MessageChannelAdapter`),核心邏輯不依賴特定平台,未來可從 Telegram 換到 LINE 而不重寫。

### 決策 2:資料外傳聲明(必須讓公司知情)

照片與語音「會」傳送至 AI 服務商 API(Claude API 做影像分析、Whisper 或同級服務做語音轉文字)。原規格「不對外發送任何資料」與 AI 功能矛盾,修正為:

- 除 AI API 之外,不傳輸資料至任何第三方
- 不公開發布任何資料
- 原始檔案永久保存於本機,不刪除
- 開發前需確認公司對工程照片送外部 API 無資安疑慮

---

## 三、技術要求

- Node.js + TypeScript
- SQLite(單檔資料庫,免架伺服器)
- 照片存 `uploads/photos/`、語音存 `uploads/voices/`,以 `日期/訊息ID` 命名,不刪原始檔
- Telegram 採 long polling(免公開網址);若選 LINE 則用 webhook + Cloudflare Tunnel
- AI 分析:Claude API(影像+文字);語音轉文字:Whisper API
- 所有程式碼加繁體中文註解
- 環境變數管理 API key(提供 `.env.example`)
- 24 小時運行環境:NAS、小主機或雲端 VM 擇一(MVP 階段一般電腦掛著跑即可)

## 四、分版開發(嚴格依序,前一版驗證通過才做下一版)

### V0 — 驗證 AI 準確率(目標 1 週)

1. Bot 接收:文字、照片、照片 caption
2. 照片下載保存,讀取 EXIF(拍攝時間,有 GPS 則記錄)
3. **相簿處理**:同一則訊息傳多張照片(media group)視為同一筆紀錄,不拆成多筆
4. AI 分析照片+文字,輸出結構化 JSON(欄位見第六節)
5. **追問機制**:必填欄位(棟別/樓層/工種)缺漏時,Bot 回覆追問,不硬猜
6. 寫入 SQLite,**記錄回報人**(平台 user id + 顯示名稱)
7. Bot 回覆整理結果,附「✅ 正確 / ✏️ 修改」按鈕(inline keyboard),人工確認後才定案

**V0 驗收**:連續 5 個工作天實際使用,統計 AI 欄位修正率,< 15% 才進 V1。

### V1 — 後台與報表(目標 1-2 週)

8. 語音訊息:下載保存 → Whisper 轉文字 → 併入 AI 分析
9. 簡單後台網頁(本機運行即可):紀錄列表、查看照片、修改欄位與狀態
10. 狀態流:`待確認 → 待改善 → 已改善 → 結案` 與 `不列管`
11. Excel 匯出,可依日期、建案、棟別、樓層、工種、狀態篩選

### V2 — 缺失追蹤閉環(目標 1 週)

12. **改善前後照片關聯**(缺失追蹤的核心):
    - 回報改善時,Bot 列出該回報人/該位置的未結案缺失供選擇,或在後台手動關聯
    - 每筆缺失紀錄可掛多張「改善後照片」,形成前後對照
13. 逾期提醒:缺失超過 N 天未改善,Bot 主動推播提醒(LINE 須注意推播額度)
14. 日報自動生成:每日彙整當日紀錄與未結案缺失,輸出 Excel/PDF

## 五、使用流程(V0)

```
工地人員:[傳 3 張照片] + caption「A棟5F 泥作 牆面空心 需打除重做」
Bot:已收到 3 張照片,AI 整理如下:
     建案:○○案|棟別:A|樓層:5F|工種:泥作
     類型:缺失|嚴重度:中
     描述:牆面空心,需打除重做
     [✅ 正確] [✏️ 修改]
工地人員:點 ✅
Bot:已建檔,缺失編號 #2026-0605-003,狀態:待改善
```

照片沒附說明時:

```
工地人員:[只傳照片]
Bot:請補充位置與工種(例:A棟5F 泥作),或語音說明。
```

## 六、AI 分析輸出 JSON

```json
{
  "project_name": "建案名稱(可由群組名稱預設)",
  "record_type": "缺失 | 一般記錄 | 改善回報",
  "building": "棟別",
  "floor": "樓層",
  "area": "區域/部位",
  "trade": "工種(見分類字典)",
  "issue_description": "缺失/記錄描述",
  "severity": "高 | 中 | 低 | 不適用",
  "suggestion": "改善建議",
  "status": "待確認",
  "reporter_id": "回報人平台ID",
  "reporter_name": "回報人顯示名稱",
  "source_message_id": "來源訊息ID",
  "photo_paths": ["uploads/photos/..."],
  "taken_at": "EXIF拍攝時間",
  "missing_fields": ["AI無法判定的欄位,觸發追問"]
}
```

**分類字典**(開發前由使用者提供,作為 AI prompt 的 few-shot 範例):
- 工種清單:泥作、鋼筋、模板、水電、油漆、防水、門窗、磁磚…(依公司實際工項定義)
- 嚴重度判定標準:各級至少 3 個實例描述
- 建議先人工標註 50-100 張歷史照片作為校準樣本

## 七、資料庫 Schema(核心表)

```sql
-- 紀錄主表
records (
  id, project_name, record_type, building, floor, area, trade,
  issue_description, severity, suggestion, status,
  reporter_id, reporter_name, source_message_id,
  taken_at, created_at, updated_at
)

-- 照片表(一筆紀錄多張照片;phase 區分改善前/後)
photos (
  id, record_id, file_path, phase,  -- phase: before | after
  exif_taken_at, created_at
)

-- 狀態異動歷程(究責與稽核用)
status_logs (
  id, record_id, from_status, to_status, changed_by, changed_at
)
```

## 八、成本估算

- Claude API 影像分析:每張約 NT$0.3-1,日均 200 張 → 月約 NT$2,000-6,000
- Whisper 語音轉文字:每分鐘約 NT$0.2,佔比極低
- 主機:NAS/既有電腦 = 0;雲端 VM 約 NT$300-600/月

## 九、風險與對策

| 風險 | 對策 |
|---|---|
| 通道選錯導致沒人用 | 決策 1 先確認;adapter 架構保留換道空間 |
| AI 分類不準 | 必填欄位靠人工說明+追問,AI 只做輔助結構化;V0 先驗證再擴 |
| 工程資料送外部 API 的資安疑慮 | 決策 2 先取得公司同意;敏感案場可關閉 AI 改純人工欄位 |
| 缺失改善無法勾稽 | V2 前後照片關聯 + 狀態歷程表 |
| 一開始做太大做不完 | 嚴格分版,V0 驗收未過不進 V1 |

## 十、給 AI 開發工具的指示

請先建立:專案結構、README、資料庫 schema(migration 檔)、`.env.example`,經確認後再開始實作 V0。每完成一個功能點先跑通再做下一個,不要一次產出全部程式碼。
