# 設計簡報：工地照片歸檔系統 — 管理後台改版

> 用途：貼給 Claude Design（或任何 UI 設計工具）做視覺改版用。
> 本檔只描述「畫面長相、資料、互動」，不含任何真實個資；所有工地名／地址／人名／編號皆為示意假資料。
> 目標讀者＝設計工具。請依本檔產出高品質視覺稿。

---

## 0. 一句話定位

這是一套「工地巡檢照片自動歸檔系統」的**內部管理後台**。現場人員用 Telegram 傳照片＋文字／錄音，系統自動判斷屬於哪個工地、編號歸檔；這個後台讓管理者**瀏覽、核對、修正、開會報告**這些紀錄。

目前後台功能完整但視覺陽春（深藍導覽列＋淺灰底＋藍色 accent＋卡片／表格）。**這次只改視覺與排版，不改功能與資訊架構。**

---

## 1. 設計目標（依重要度）

1. **沉穩務實的「工程感」**——使用者是工地主任與老闆，年紀偏長，重點是**好讀、對比清楚、資訊一眼抓得到**，不要花俏。
2. **照片是主角**——這是照片歸檔系統，縮圖／代表照片的呈現要漂亮、整齊、放大好看。
3. **桌機＋手機都要（RWD）**——桌機是日常核對主場；老闆／主任會用手機看「報告頁」。兩種寬度都要好用。
4. **狀態與警示一眼可辨**——「待改善」「判不出（_inbox）」這類需要處理的項目要醒目。
5. **報告頁要能列印／存 PDF**——拿去開會、跟老闆報告。

### 風格方向：工程感／沉穩務實
- 建議色系：工地藍灰為主調（沉穩、專業）＋一個安全橘／琥珀作為「需要注意」的強調色。可參考工程／施工管理軟體的氣質，但**乾淨現代**、避免廉價感。
- 對比清楚、字級偏大、留白足夠；長者也能輕鬆讀。
- 介面語言：**繁體中文**。字型需含繁中（如 Noto Sans TC／思源黑體／系統 Microsoft JhengHei），數字與編號建議等寬字（編號是 `A001-20260615-001` 這種格式）。

---

## 2. 使用者與情境

| 角色 | 用途 | 主要裝置 |
|---|---|---|
| 工地主任 | 每日核對：今天幾筆、有沒有掉進「判不出」、判定對不對；幫判不出的照片人工指定工地；改狀態、補備註 | 桌機為主，現場用手機 |
| 老闆 | 看「報告頁」掌握各工地進度，開會／簡報 | 手機、桌機、列印 |

- 後台只綁本機（127.0.0.1），單人或少數人使用，**不需要登入頁、不需要多帳號權限管理**。
- 資料量小（一天數筆到數十筆）；不需要複雜的分頁、虛擬列表。

---

## 3. 資訊架構（全域）

頂部全域導覽，三個主分頁固定可切換：

```
🗂 工地照片歸檔 — 管理後台      [儀表板]  [紀錄列表]  [報告]
────────────────────────────────────────────────
（每頁標題列下方有一條「摘要列」：顯示該頁關鍵數字，例：共 N 筆／_inbox N ⚠️）
```

四個畫面：
1. **儀表板** `/dashboard` — 總覽統計
2. **紀錄列表** `/records` — 可篩選的紀錄卡片清單（列表的點進去是↓）
3. **單筆詳細** `/records/{id}` — 一筆紀錄的完整內容＋可編輯動作
4. **報告頁** `/report` — 按工地分區的圖文報告，給開會／報告用，可列印

---

## 4. 共用元件（四頁都會用到，請統一設計）

### 4-1. 狀態 badge（共 4 種，顏色要可區分且語意正確）
| 狀態 | 語意 | 現用色感（可重新設計） |
|---|---|---|
| `待確認` | 剛歸檔、等人確認 | 黃／待辦 |
| `待改善` | 已確認、現場有問題待修 | 藍／進行中（**最常被追蹤**） |
| `已完成` | 問題已處理 | 綠 |
| `已結案` | 整件收掉 | 綠／灰 |

### 4-2. 判定方式 badge（系統怎麼判出工地的，共 6 種，視覺次要、可低調）
`手動代碼`、`照片 GPS`、`TG 位置`、`近期上下文`、`按鈕選擇`、`判不出`

### 4-3. 工地標示
- 正常工地：顯示「`代碼` 名稱」，例：`A001 信義示意案`（代碼建議等寬字）。
- 判不出：顯示醒目的「⚠️ _inbox 判不出」，用強調色（橘／琥珀），提示這筆需要人工處理。

### 4-4. 媒體縮圖
- 照片縮圖：正方形裁切、圓角、深色底（避免透明 PNG 破圖）。
- 錄音：用播放器或「🎤 錄音」標示。
- 文件 📄 / 無法預覽（如 HEIC）：用佔位卡，顯示「HEIC 無法預覽」或「⚠️ 檔案不存在」。

### 4-5. 卡片／表格／空狀態
- 卡片：白底、細邊框、圓角；hover 有回饋。
- 空狀態：例「沒有符合條件的紀錄」「這段期間沒有任何紀錄」，要友善置中。

## 5. 逐頁規格

### 頁 A：儀表板 `/dashboard`

**目的**：開機第一眼，掌握全局＋揪出待處理。

**內容（由上到下）**：
1. **摘要列**：全部 N 筆 ｜ _inbox N ⚠️
2. **統計卡片列**（橫向卡片）：
   - 全部紀錄（數字）
   - ⚠️ _inbox 待歸檔（數字，**有值時要警示色**）
   - 各狀態數量（待確認／待改善／已完成…各一張卡，帶該狀態的 badge）
3. **_inbox 警示橫幅**（只有 _inbox > 0 時出現）：「⚠️ 有 N 筆判不出工地，前往人工歸檔 →」
4. **最近 7 天收件**：7 列長條圖（每列：日期＋橫條＋筆數＋「查看」連結到當天列表）。**橫條長度依當日筆數**。
5. **各工地**：每個工地一列（代碼、名稱、累計筆數、最後收件時間），點名稱進該工地列表。
6. **判定方式分布**：每種判定方式一列（名稱＋筆數＋查看連結），依筆數多到少。

**設計重點**：卡片數字要大、好掃；_inbox 卡片與警示橫幅要跳出來；長條圖簡潔即可（不用花俏圖表庫，水平 bar 就好）。

---

### 頁 B：紀錄列表 `/records`

**目的**：日常核對主畫面，可多條件篩選後逐筆檢視。

**內容**：
1. **摘要列**：符合 N 筆 ｜ 已歸檔 N ｜ _inbox N ⚠️ ｜ 待確認 N
2. **篩選列**（一排，桌機橫向、手機可收合）：
   - 日期（收件日，date picker）
   - 工地（下拉：全部／⚠️ _inbox 判不出／各工地）
   - 狀態（下拉：全部／待確認／待改善／…）
   - 判定方式（下拉：全部／6 種）
   - 「篩選」按鈕 ＋「清除條件」連結
3. **紀錄卡片清單**（新的在上），每張卡片一筆，顯示：
   - 第一行：**紀錄編號**（等寬粗體）｜ 工地標示 ｜ 狀態 badge ｜ 判定方式 badge ｜（靠右）N 件 · MM/DD HH:MM
   - 第二行：回報人（如「王示意（未具名）」）
   - 第三行（若有）：文字備註（灰底小區塊，可換行）
   - 整張卡片可點 → 進詳細頁
4. **空狀態**：「沒有符合條件的紀錄。」

**設計重點**：卡片密度適中、編號要顯眼、各 badge 對齊整齊；手機版卡片直向堆疊、篩選列要好按。

---

### 頁 C：單筆詳細 `/records/{id}`

**目的**：看一筆的全貌，並就地修正（這頁有**寫入動作**）。

**內容**：
1. 返回列表連結（← 回列表）
2. **基本資料表**（key–value 兩欄）：
   - 紀錄編號（粗體等寬）
   - 工地（顯示目前工地或「⚠️ _inbox 判不出」）＋**就地操作**：一個工地下拉＋「改工地 / 指定工地（人工歸檔）」按鈕（送出前會跳確認框：照片會搬到新工地資料夾）
   - 狀態（目前狀態 badge）＋**就地操作**：「改為 待改善 / 已完成 / …」按鈕群（除目前狀態外的選項）
   - 判定方式
   - 回報人
   - 收件時間（標注「歸檔日期依據」）
   - 拍攝時間（EXIF，可能「（無 EXIF）」）
   - GPS（座標或「（無）」）
   - 文字備註：**可編輯 textarea ＋「儲存備註」按鈕**（提示：儲存會同步重寫歸檔目錄的 metadata.json / text.txt）
   - 來源（如 telegram；可能標「（相簿合併）」）
3. **媒體區**「媒體（N 件）」：照片用 `<img>`、錄音用 `<audio controls>`、不可預覽用佔位卡。每件下方小字：檔名、類型 badge（🖼 照片／🎤 錄音／📄 文件）、EXIF 時間／📍。
4. **狀態歷程**「狀態歷程」：時間軸／表格，每列：時間、`原狀態 → 新狀態`、由誰改（如「後台網頁」「王示意」）。

**設計重點**：這頁是「閱讀＋就地編輯」混合，編輯控制項要明顯但不喧賓奪主；表單按鈕要清楚是可點的動作。手機版 key-value 表改為上下堆疊。

---

### 頁 D：報告頁 `/report`（最值得花心思，會拿去開會／列印）

**目的**：按工地分區的圖文總覽，挑代表照片，給老闆／開會看；可列印或存 PDF。

**內容**：
1. **摘要列**：期間 `2026-06-09 ~ 2026-06-15` ｜ 共 N 筆 ｜ 待改善 N ｜ _inbox N ⚠️
2. **控制列**（列印時自動隱藏）：
   - 期間下拉：今天／近 7 天／近 14 天／近 30 天／自訂區間
   - 自訂時出現「起」「迄」兩個 date picker
   - 「套用」按鈕
   - 「🖨 列印 / 存 PDF」按鈕
3. **按工地分區**，每個工地一個區塊（section，依筆數多到少，**_inbox 永遠排最後**）：
   - 區塊標題：「`代碼` 工地名稱」＋（靠右）N 筆 · 最後 MM/DD HH:MM
   - 狀態分布：一排 badge（待確認 N／待改善 N／已完成 N…，固定順序）
   - **代表照片縮圖牆**（每工地最多 8 張，正方形）：
     - 縮圖右下角若有「🎤」表示該照片所屬紀錄有錄音
     - 點縮圖 → 開**頁內放大層（lightbox）**
   - 無可預覽照片時：「（此區間無可預覽照片）」
4. **放大層 lightbox**（點縮圖開啟）：
   - 置中大圖（深色半透明背景遮罩）
   - 圖下方文字註解（**字級要大，現用 24px**，內容是「編號　備註」）
   - 若該紀錄有錄音：一顆「🎤 播放錄音」按鈕，按了才出現播放器
   - 「關閉 ✕」按鈕；點背景或按 Esc 也關閉
5. **列印樣式（@media print）**：隱藏導覽、控制列、頁尾、返回鈕、lightbox；背景轉白；**每個工地區塊不跨頁切斷**（break-inside: avoid）。
6. **空狀態**：「這段期間（…~…）沒有任何紀錄。」

**設計重點**：這頁要「漂亮到可以直接給老闆看」。照片牆排版要整齊大方；列印出來要乾淨、像一份正式日報；手機上也要能順順看完（縮圖牆自動換行、lightbox 全螢幕）。

## 7. 硬性約束（請務必遵守）

- **繁體中文**介面，字型含繁中。
- **桌機＋手機 RWD**：桌機寬版（內容區約 1100px 置中），手機單欄。
- **報告頁可列印／存 PDF**：要有乾淨的列印樣式。
- 不需要：登入頁、權限管理、深色／淺色切換（除非你想加分，但主視覺以淺色沉穩為準）、複雜圖表庫。
- 照片縮圖／放大層是體驗核心，請優先打磨。

## 8. 交付建議

請產出這 4 個畫面的高品質視覺稿（桌機版必做，報告頁與列表頁的手機版也請一併示意），含：
- 一套設計 token（色票、字級、間距、圓角、陰影）
- 共用元件（狀態 badge ×4、判定 badge、卡片、縮圖、lightbox）
- 四頁的完整版面

> 備註（給回貼實作的人看，非設計工具需要）：實際後端是單檔 Node http 伺服器、用字串模板吐 HTML、CSS 全內嵌、無前端框架。視覺稿用什麼技術都行，但最終會「人工搬」回 vanilla HTML/CSS，請避免依賴難以轉譯的重型互動。
