# PROJECT_OVERVIEW — 工地照片歸檔系統

## 專案目的
讓建設公司工地人員在 Telegram 群組直接傳「照片＋文字/語音說明」，系統自動把照片歸檔到對應工地資料夾，結構化為工地紀錄與缺失追蹤，並提供後台查詢、狀態管理與 Excel 匯出。核心價值是讓缺失從「回報 → 改善 → 銷案」全程可追蹤、不漏件，取代人工整理。

## 業務背景
- 需求方：建設公司總經理特助（使用者的同學）。
- 痛點：每日大量工地照片（品質記錄、缺失待改、改善完成）人工整理耗時、缺失追蹤易漏件。
- 使用規模：初期 1–3 人（特助自用）→ 若要全工地人員回報，須改走 LINE（已用 adapter 介面預留）。

## 技術選型
- 後端：Node.js + TypeScript。
- 資料庫：SQLite（better-sqlite3，單檔免架伺服器）。
- 訊息通道：**Telegram Bot，long polling（免公開網址）**；訊息接收層做 `MessageChannelAdapter` 介面，核心邏輯不綁平台，未來可換 LINE（LINE 須 webhook + 公開 HTTPS）。
- EXIF 解析：exifr（需支援 HEIC）。
- AI（V1+ 才上）：Claude API 做影像分析、Whisper 做語音轉文字；介面預留 `AIAnalyzer`，V0 不接。
- 運行環境：24 小時掛機（NAS DS1517+ / 小主機 / 雲端 VM 擇一；MVP 階段一般電腦掛著跑即可）。
- 規格全文見 `docs/PRD_v2.md`、開發脈絡見 `docs/HANDOFF.md`（兩者衝突時，以後續對話確認的修正為準）。

## 北極星
缺失全程閉環可追蹤、零漏件：工地人員拍照＋說明回報一筆 < 30 秒，AI 欄位辨識準確率 > 85%（以人工修正率回推），每筆缺失「回報 → 改善 → 銷案」狀態與前後照片都勾稽得起來。

## 避雷區
- **Telegram photo 會壓掉 EXIF** → 重要照片走 document 上傳保留原圖；EXIF 拍攝時間另存欄位。
- **判不出工地不准硬猜** → 一律進 `data/_inbox/` 暫存 + 按鈕詢問使用者，五層優先序跑完才定案。
- **不一次產出全部程式碼** → 嚴格分版，每個功能點先跑通再做下一個；V0 驗收未過不進 V1。
- **工程照片送外部 AI API 有資安疑慮** → 開發前須取得公司知情同意；敏感案場可關 AI 改純人工欄位。除 AI API 外不傳任何第三方。
- **HEIC EXIF** 用 exifr，別用不支援 HEIC 的舊套件踩坑。
- **照片/個資絕不進 Git**（`.gitignore` 已擋）→ 原始檔永久保存於本機/NAS，不刪。
